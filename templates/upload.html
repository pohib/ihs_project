Справочник полей

name

Название правила. Если внутри есть запятые, используйте "кавычки".

source_ip / dest_ip

IP или подсети. Формат: 192.168.1.1 или 10.0.0.0/24.

port

Число (1-65535) или диапазон (например, 80-443). Оставьте пустым для ANY.

action

Только два значения: ALLOW или DENY.

Образец CSV Excel / Notepad
# Заголовок обязателен

Rule1,192.168.1.1,0.0.0.0/0,443,TCP,ALLOW
"DNS",8.8.8.8,8.8.4.4,53,UDP,ALLOW

Справочник UserGate

Версии: 5.x, 6.x, 7.x (JSON Export)

1
Навигация

Раздел «Политики сети»
→ вкладка «Межсетевой экран».

2
Выбор данных

Выделите нужные правила и нажмите «Экспорт». Выберите «Экспорт с зависимыми объектами».

3
Формат

Тип файла: JSON.
Обязательно включите вложенные объекты.

Логика обработки полей

Поле в UserGate Действие системы
src_ips / dst_ips Раскрывает списки IP, диапазоны и CIDR. Группы объектов анализируются рекурсивно.
services Извлекает L4-протокол (TCP/UDP) и порты. Если указан "AppID", правило помечается для спец. проверки.
action "pass" → ALLOW, "drop/deny" → DENY.
JSON Structure
{
  "rules": [
    {
      "name": "Access to DB",
      "enabled": true,
      "src_ips": ["172.16.10.0/24"],
      "dst_ips": ["10.0.0.50"],
      "services": [
        { "proto": "tcp", "port": "5432" }
      ]
    }
  ]
}

Критически важно при импорте

Отключенные правила: Если в JSON поле "enabled": false, то правило будет полностью проигнорировано при анализе конфликтов.

L7-фильтрация: Параметры Content Filtering и SSL Inspection не учитываются, так как анализ сфокусирован на уровне L3/L4.

Зоны (Zones): Парсер сопоставляет Source/Destination зоны. Убедитесь, что названия зон в файле соответствуют вашей сетевой топологии.

Предел объектов: При наличии более 5000 объектов адресов в одном файле время первичной обработки может увеличиться до 30 секунд.

Справочник АПКШ «Континент» версии 3.9

Работа с XML конфигурацией ЦУС (Центр Управления Сетью)

Инструкция по выгрузке из Менеджера Конфигурации

1

Запустите «Программу управления ЦУС» и авторизуйтесь под учетной записью с правами администратора конфигурации.

2

Выберите необходимый КШ (Криптошлюз) в дереве устройств. Перейдите в меню «Конфигурация» → «Экспорт».

3

В появившемся окне выберите «Правила фильтрации». Сохраните полученный файл с расширением .xml.

Важно: Не путайте с экспортом ключей или системного журнала.
Маппинг XML-аттрибутов
Идентификация правила

Система считывает аттрибут Description. Если он пуст, правилу присваивается индексный номер Rule_#ID.

Действия (Action)

Permit интерпретируется как ALLOW, остальные (Deny, Reject) как DENY.

<!-- Структура объекта -->
<FilterRule Description="WAN_Access" ...>
  <Source>
    <IPRange From="10.0.0.1" To="10.0.0.255"/>
  </Source>
  <Service Protocol="6" Port="80"/>
</FilterRule>

Особенности версии 3.9

Диапазоны IP (IPRange) Система анализирует весь интервал адресов (From/To). Обратные диапазоны помечаются как ошибка.

Числовые протоколы В Континенте протоколы часто указаны ID (6=TCP, 17=UDP, 1=ICMP). Наша система выполняет обратный маппинг в человекочитаемый вид.

Стелс-правила Правила, созданные системой автоматически (скрытые), не экспортируются в XML и не участвуют в анализе избыточности.

Кодировка XML должен быть сохранен в кодировке UTF-8 или Windows-1251. При проблемах с кириллицей проверьте заголовок файла.

Важно для анализа конфликтов: Континент 3.9 применяет правила сверху вниз до первого совпадения. Наш алгоритм учитывает эту последовательность при поиске «затененных» (Shadowed) правил.

ViPNet Administrator (ЦУС)

Работа с XML-экспортом правил фильтрации

Где взять файл экспорта?

ШАГ 1

В ЦУС перейдите в раздел «Прикладные задачи» → «Межсетевой экран».

ШАГ 2

Выберите нужный сетевой узел (Координатор) или группу. Откройте вкладку «Правила фильтрации открытого трафика».

ШАГ 3

Нажмите ПКМ на списке правил и выберите «Экспорт...». Укажите тип файла XML и кодировку UTF-8.

Поддерживаемые объекты
  • Транзитные правила (Forward)
  • Локальные правила (Input)
  • Групповые объекты адресов

Технические детали парсинга

<!-- Система ищет следующие теги -->
<FirewallRule Name="ERP_Access" Action="Allow">
  <Source>192.168.10.5</Source>
  <Destination>10.0.0.1/32</Destination>
  <Protocol>TCP</Protocol>
  <Port>1433</Port>
</FirewallRule>
Маскировка

Правила с действием "Маскировать" (NAT) учитываются как ALLOW, но помечаются флагом NAT для корректного поиска дубликатов.

Протоколы

Если в поле Protocol указано "Любой" или "ANY", система будет сравнивать правило со всеми портами других правил.

Важные нюансы работы ЦУС

Открытый vs Туннелируемый трафик

Парсер предназначен для анализа открытого (нешифрованного) трафика. Правила для трафика внутри VPN-туннелей ViPNet (между узлами сети) выгружаются в другие секции XML и в данном модуле не анализируются.

Пользовательские объекты

Если в качестве источника или назначения используются «Сетевые объекты», созданные вручную, убедитесь, что при экспорте XML опция «Раскрывать сетевые объекты» включена, иначе вместо IP-адресов система увидит только их имена.

Приоритет (Order)

В ViPNet правила могут иметь разный приоритет в зависимости от того, являются они Сетевыми или Индивидуальными. При анализе мы автоматически выстраиваем их в правильном порядке исполнения (Сначала Сетевые, затем Индивидуальные).

Кодировка XML

При экспорте из ЦУС старых версий может использоваться кодировка windows-1251. Наш загрузчик поддерживает её, но для стабильности рекомендуем пересохранять файл в UTF-8.
